# FPS Server v1.3 — 2vs2 3D FPS 게임 서버 프로그래밍 학기 프로젝트 기획서

> **5주 / 2인 / 자체 제작 서버 / 학기 산출물 + 방학 포트폴리오 보강**

---

## 변경 이력

- **v1.0** — 초안 (2vs2 / TCP+UDP / 5주)
- **v1.1** — WBS v1과 정합성 맞춤 (학기/방학 분리, 비주얼 정책 명시, Walking Skeleton 명시)
- **v1.2** — MVP 핵심을 **"자동 매칭 기반 TCP/UDP 멀티플레이 풀 사이클"** 로 재정의
  - 장르 확정: **3D 1인칭 FPS**
  - **자동 매칭 시스템** (단일 매치, FIFO 큐) 도입
  - 로비 / 방 / Ready / 방장 시작 → **4주차 확장 또는 방학 보강**
  - 라운드 승리 조건 단순화 (**한 팀 전원 사망**만 MVP)
  - 매치 종료 단순화 (**2라운드 선취**만 MVP) + **한 팀 전원 이탈 시 즉시 종료** 추가
  - **ESC 종료 모달 → 4주차 확장**, MVP는 **TCP 끊김 감지 → 사망 처리**로 대체
  - **정확한 hitscan (ray-sphere)** MVP 포함
  - **이동 + 점프** 포함 (점프 시 서버 중력 시뮬레이션 필요)
- **v1.3** — 정합성 정정
  - **Unity 2022 LTS** 명시 (v1.2의 "Unity 6" 표기 정정 — 팀 합의는 2022 LTS)
  - **탄약 / 재장전 → 4주차 확장으로 명시 분류** (MVP 사격 사이클은 무한 탄약)
  - **틱레이트 단계적 도입 강조** — 1~2주차는 "받자마자 처리" 단순 모델, 3주차 UDP 도입과 함께 정식 30Hz/20Hz 틱 루프 도입
  - 시뮬/스냅샷 빈도 분리에 대한 짧은 설명 추가

## 1. 프로젝트 개요

| 항목 | 내용 |
|---|---|
| 가제 | `2vs2 FPS Arena` |
| **장르** | **2vs2 팀 라운드제 3D 1인칭 슈팅 (FPS) — 멀티플레이 프로토타입** |
| **플랫폼** | **Windows PC** (Unity 2022 LTS 3D 클라이언트 + .NET 8 콘솔 서버) |
| 기간 | 5주 |
| 인원 | 2명 (서버 1 / 클라 1) |
| 목적 | ① 게임서버프로그래밍 수업 학기 프로젝트<br>② 게임 서버 직무 포트폴리오 산출물 (방학 보강 포함) |
| 리포지토리 | Cuyacuya/2vs2_Shooter_Server (모노리포) |

### 한 줄 소개
4명이 자동 매칭으로 모이면 시작되는 **2vs2 3D FPS**. 먼저 2라운드를 가져간 팀이 승리하는 라운드 기반 슈팅 프로토타입.

### 프로젝트가 증명하려는 것
**"상용 네트워크 솔루션(Mirror, Photon, NGO 등) 없이, 직접 짠 C# 서버로 자동 매칭 → TCP/UDP 하이브리드 멀티플레이 풀 사이클을 동작시킬 수 있다."**

특히 다음을 직접 구현하여 학습/포트폴리오 가치 확보:
- 직접 설계한 패킷 프로토콜 (TCP 프레이밍, UDP 세션 매칭)
- 직접 만든 **자동 매칭 큐** (FIFO, 단일 매치)
- **Server Authoritative** 게임 상태 머신 (이동, 점프, hitscan)
- **TCP/UDP 하이브리드 통신** (신뢰성 vs 성능 분리)
- **봇 더미 클라이언트**로 4인 환경 자동 테스트

## 3. 기술 스택

### 서버
- **언어/런타임**: C# / .NET 8 콘솔 애플리케이션
- **소켓**: `System.Net.Sockets` (TcpListener / UdpClient + 비동기 I/O)
  - 여유 시 `SocketAsyncEventArgs` 기반 리팩토링 (방학 또는 C++ 리메이크)
- **직렬화**: 직접 정의한 바이너리 패킷 (헤더 + payload)
- **로깅**: 콘솔 + 파일
- **저장소**: 메모리 only (DB 없음)

### 클라이언트
- **엔진**: **Unity 2022 LTS** (3D, 정확한 마이너 버전 팀과 통일)
- **공유 코드**: 서버와 동일한 패킷 정의 프로젝트(`Shared`) 공유
  - 1주차: 수동 복사
  - 2주차 이후: 자동화 검토 (DLL 빌드 또는 Symlink)

### 비주얼 정책 (학기 동안)

프로토타입 우선 원칙. 5주 안에 시스템 검증이 목표이므로 비주얼은 최소화.

| 요소 | 학기 (5주) | 방학 폴리싱 |
|---|---|---|
| 캐릭터 | 캡슐 + 시점 방향 표시 (작은 큐브) | 모델/리깅 |
| 시점 | 1인칭 (캡슐 머리 위 카메라) | 그대로 또는 3인칭 옵션 |
| 무기/사격 | Line Renderer (0.1초 사격선) + 조준점(Crosshair) | 머즐 플래시, 파티클 |

## 4. 게임 규칙

### 기본
| 항목 | 값 |
|---|---|
| 인원 | 2vs2 (4인 고정) |
| 시점 | **1인칭** |
| 맵 | 1개 (좌우 대칭, 평면 + 큐브 엄폐물) |
| 무기 | hitscan 권총 1종 (**MVP는 무한 탄약**, 탄약/재장전은 4주차 확장) |
| 이동 | WASD + Space (점프) + 마우스 룩 |
| 경기 방식 | 3라운드 2선승 |

### 플레이어 상태
- **HP 100**, 사격 시 **데미지 25 고정** (4발 = 사망)
- 헤드샷/바디샷 구분 없음
- 라운드 중 리스폰 없음
- 라운드 시작 시 HP/위치/사망 플래그 리셋

### 사격 (Hitscan)
**클라**:
- 마우스 클릭 시 → 카메라 위치 + 카메라 forward 방향을 `C_Fire` 패킷에 담아 송신
- Line Renderer로 0.1초간 사격선 표시 (시각 효과만)

**서버**:
- 발사자 위치 + 방향으로 ray 생성
- 4명 캐릭터를 **구(sphere) 1개로 단순 근사** → ray-sphere 충돌 검사
- 가장 가까운 적 1명에 데미지 25
- (자기 팀 ray 통과 — 팀킬 없음, MVP 단순화)

> **MVP 단순화**: 무한 탄약, 무한 연사 가능 (입력 빈도만큼 사격). 탄약 카운터/재장전/연사 속도 제한은 4주차 확장.

### 이동 / 점프
**클라**:
- WASD → 이동 입력
- Space → 점프 입력 (지면 접촉 시에만)
- 마우스 룩 → 시점 회전 (yaw/pitch)
- 위 모두 `C_Input` 패킷으로 서버에 송신 (UDP)

**서버**:
- 입력 받아 위치 갱신
- **점프**: 단순 중력 시뮬레이션
  - 매 틱: `velocity.y -= 9.8 * dt`
  - 점프 입력 + 지면 접촉 시: `velocity.y = +5.0` (초기 상승 속도)
  - `y <= 0`이면 지면 정지 (`y=0, velocity.y=0, isGrounded=true`)
  - 더블점프 / 벽 점프 / 슬라이딩 없음
- 이동 속도 = 5.0 m/s (튜닝)

### 라운드 승리 판정 (MVP)
- **한 팀 전원 사망 → 다른 팀 라운드 승리, 점수 +1**
- (MVP 제외: 시간 제한, 생존자 수 비교, 팀 총 HP 비교, 무승부 처리)

### 매치 종료 판정 (MVP)
- **한 팀이 2라운드 선취** → 매치 종료, 해당 팀 승리
- **라운드 시작 시 한 팀 살아있는 인원 0명** (전원 이탈/사망 상태) → 즉시 매치 종료, 남은 팀 승리

### 중도 이탈
- 인게임 중 1명 이탈 (TCP 끊김 또는 명시적 종료)
  → **즉시 사망 처리, 해당 매치 끝까지 부활 안 함, 라운드 진행**
- 한 팀 전원 이탈 → 위 매치 종료 조건 발동
- (MVP 제외: ESC 종료 확인 모달 — 끊김만 감지)

## 5. MVP 게임 흐름

### 5-1. 자동 매칭 흐름

```
[Main 씬]
  ↓ 닉네임 입력 + "접속" 버튼
  ↓ TCP 서버 접속
  ↓ C_Login 송신 (닉네임 포함)
  ↓
[서버: 매칭 슬롯 검사]
  ├─ 비어있음 또는 대기 중 (0~3명)
  │   → S_LoginResult(success=true, sessionToken, slotNumber=N)
  │   → 매칭 대기열 등록
  │   → 대기 중인 모든 클라에 S_MatchingStatus(N/4) 브로드캐스트
  │
  └─ 매치 진행 중
      → S_LoginResult(success=false, reason="MATCH_IN_PROGRESS")
      → TCP 연결 종료
  ↓
[Main 씬 — 매칭 대기 UI]
  "매칭 대기 중 N/4" 표시 + "취소" 버튼
  ↓
[서버: 4명 모이면]
  - 입장 순서로 팀 배정 (1,3=Red / 2,4=Blue)
  - PlayerState 4개 초기화 (HP, 위치, 팀)
  - 4명 모두에게 S_GameStart 송신
  ↓
[Game 씬 전환]
  - 서버가 준 팀/플레이어 정보로 캡슐 4개 생성
  - 자기 캡슐에 1인칭 카메라 부착
  - WASD + Space + 마우스 입력 → C_Input (UDP)
  - 마우스 클릭 → C_Fire (TCP)
  - 서버 S_Snapshot 받아 4명 위치/HP 갱신 (UDP)
  ↓
[라운드 진행]
  서버: 입력 처리 → 위치/사격/HP/사망 갱신 → 스냅샷 송신
  한 팀 전원 사망 검사 → S_RoundEnd(승리팀) → 점수 +1
  ↓
[다음 라운드 또는 매치 종료]
  - 양 팀 모두 2승 미만 → S_RoundStart(라운드N+1) → HP/위치/사망 리셋
  - 한 팀 2승 또는 한 팀 전원 이탈 → S_MatchEnd(승리팀, 최종 점수)
  ↓
[결과 화면]
  Win/Lose + 최종 스코어 + "메인으로" 버튼
  ↓
[Main 씬 복귀]
  TCP 끊고 다시 연결 가능 상태
```

### 5-2. 자동 매칭 정책

| 항목 | 정책 |
|---|---|
| 매치 동시 진행 수 | **단일 매치 (1개)** |
| 5번째 접속자 처리 | **로그인 단계에서 거부** (S_LoginResult.success=false) |
| 매칭 대기 취소 | TCP 끊으면 자동으로 대기열에서 제거 |
| 매치 종료 후 | 슬롯 비우고 다시 대기 모드 진입, 다음 매치 가능 |

**MVP에서 의도적으로 제외:**
- 로비 / 방 목록 / 방 생성 / 방 입장
- Ready 토글 / 방장 시작 버튼
- 매치 진행 중 신규 접속자 대기 큐 (그냥 거부)

위 항목은 **4주차 확장 기능 또는 방학 보강**으로 이동.

### 5-3. 팀 배정 (입장 순서 기준)

| 입장 순서 | 팀 |
|---|---|
| 1번째 | Red |
| 2번째 | Blue |
| 3번째 | Red |
| 4번째 | Blue |

## 6. 화면 구성

### 6-1. Main 씬 (초기 상태)
- 닉네임 입력 필드
- "접속" 버튼 → TCP 접속 + C_Login
- "종료" 버튼 → 앱 종료

### 6-2. Main 씬 (매칭 대기 상태)
- "매칭 대기 중 **N/4**" 텍스트 (서버 S_MatchingStatus 갱신)
- "취소" 버튼 → TCP 끊고 초기 상태로 복귀

### 6-3. Game 씬 (인게임)
- 1인칭 카메라 (캡슐 머리 위치)
- HP Slider (좌하단)
- 라운드 점수 Text (상단 중앙: `RED 1 - 0 BLUE`)
- 조준점(Crosshair, 화면 중앙 작은 점)
- Line Renderer (사격 시 0.1초 표시)

### 6-4. Game 씬 (라운드 종료 / 매치 종료 오버레이)
- **라운드 종료**: "Round Won/Lost" 텍스트 3초 표시 후 자동 다음 라운드
- **매치 종료**: "Victory" / "Defeat" + 최종 스코어 + "메인으로" 버튼

### MVP 제외 (확장)
- ESC 종료 확인 모달 (4주차 확장)
- **탄약 UI / 재장전 표시 (4주차 확장)**
- 미니맵 / 킬로그 / 데미지 숫자
- 채팅
- 스코어보드 (Tab 키)

## 7. 서버 아키텍처

### 권한 모델: Server Authoritative + State Synchronization
- **클라**: 입력(이동, 점프, 마우스 룩, 사격)만 송신
- **서버**: 모든 게임 상태 결정 (위치, HP, 사격 판정, 사망, 라운드/매치 상태)
- 클라는 서버 스냅샷대로 렌더링 (보간만 클라가 함)

### 시뮬레이션 / 네트워크 틱 — 단계적 도입

**3주차 이후 목표 스펙** (학기 끝의 모습):
| 항목 | 빈도 | 설명 |
|---|---|---|
| 게임 시뮬레이션 틱 | **30Hz** | 서버가 1초에 30번 입력 큐를 일괄 처리, 위치/중력/충돌 갱신 |
| 스냅샷 송신 | **20Hz** | 서버가 1초에 20번 4명 분량 상태를 클라에 브로드캐스트|
| 클라 렌더링 | 60Hz | 수신 스냅샷 사이를 Lerp 보간 |

**왜 시뮬과 스냅샷 빈도가 다른가**: 시뮬은 빠를수록 정확하지만, 스냅샷은 비싸다 (네트워크 대역폭). 4명 × 위치/HP/회전 등 = 패킷 1개 약 50~80byte, 4명에게 송신 → 인원이 늘면 N²로 폭증. 그래서 **시뮬은 빠르게(정확도 우선), 스냅샷은 그보다 느리게(대역폭 절약)** 가 표준. 클라가 받은 스냅샷 사이를 보간으로 메꾼다. (CS:GO 64/64Hz, Overwatch 60/20Hz, 학기 4인 규모는 30/20Hz로 충분)

**1~2주차 (Walking Skeleton)**: 위 정식 틱 루프를 **처음부터 짜지 않는다**. "받자마자 처리, 즉시 송신"이라는 가장 단순한 모델로 시작:

```csharp
// 1~2주차 의사코드 (받자마자 처리)
while (true) {
    var packet = await ReadPacketAsync();
    HandlePacket(packet);
    BroadcastIfNeeded();
}
```

이걸로도 4명이 움직이는 건 동작한다 (어색하지만). **TCP 위치 동기화의 한계를 직접 체감**하는 것이 학습 핵심.

**3주차 (UDP 도입과 동시에 정식 틱 루프 도입)**:

```csharp
// 3주차 의사코드 (정식 30Hz/20Hz 틱)
var sw = Stopwatch.StartNew();
while (true) {
    if (sw.ElapsedMilliseconds >= 33) {  // 30Hz = 33ms마다
        ProcessAllQueuedInputs();         // 큐에 쌓인 입력 일괄 처리
        SimulateOneTick();                // 위치/중력/사격/HP 갱신
        if (snapshotCounter++ % 1.5 == 0) {
            BroadcastSnapshot();          // 20Hz로 다운샘플
        }
        sw.Restart();
    }
}
```

UDP는 패킷 손실/순서 뒤집힘이 있으므로 틱 단위 일괄 처리가 자연스럽다. **1~2주차 한계를 본인이 느낀 후** 도입 → 학습 효과 극대화.

### 통신 구조: TCP + UDP 하이브리드
| 채널 | 용도 |
|---|---|
| **TCP (7777)** | 접속, 매칭, 사격(C_Fire), 게임 시작/종료, 라운드 시작/종료 |
| **UDP (7778)** | 인게임 입력 송신(C_Input), 상태 스냅샷 브로드캐스트(S_Snapshot) |

**세션 식별**: TCP 접속 시 발급한 세션 토큰을 UDP 패킷에도 포함해 매칭 (UDP는 비연결 — 누가 보냈는지 모름)

**메시지 분류 원칙**:
- 신뢰성 필요 + 빈도 낮음 → TCP (사격 판정, 라운드 결과 등)
- 자주 보내는 실시간 데이터 → UDP (이동 입력, 위치 스냅샷)

> 사격을 TCP로 한 이유: hitscan 1발의 결과가 게임에 큰 영향. 누락되면 안 됨. 빈도도 낮음(클릭 시).

### 서버 상태 머신 (MVP — 단순화)

```
[Idle: 0~3명 대기]
   │
   │ 4명 모임
   ↓
[Matching → 즉시 InGame: Round 1]
   │
   │ 라운드 종료 → 점수 +1
   ↓
[InGame: Round 2] ──→ [InGame: Round 3]
   │
   │ 한 팀 2승 또는 한 팀 전원 이탈
   ↓
[MatchEnd → 4명 메인 복귀, Idle 복귀]
```

> 기존 v1.1의 [Lobby] / [RoomWaiting] 상태는 4주차 확장 또는 방학으로 이동.

### 개발 전략: Walking Skeleton
기능 단위가 아닌 **사이클 단위 점진 강화**.
- 1주차부터 끝까지 한 번 통합 (어설퍼도 OK)
- 매주 통합 시점 유지
- **임시 코드 → 정식 구조로 점진 교체** (틱 루프, UDP 등이 이 패턴)

> 업계 정식 기법 (Tracer Bullet, Vertical Slice). 통합 시점을 앞당겨 통합 버그를 일찍 발견.

## 8. 패킷 프로토콜 (개요)

상세는 `docs/protocol.md`에서 관리.

### 공통 헤더 (6 byte, Little Endian)
| 필드 | 크기 | 설명 |
|---|---|---|
| length | uint16 | payload 길이 |
| packetId | uint16 | 패킷 종류 |
| sessionToken | uint16 | 세션 식별자 (UDP 매칭용) |

### 합의 사항
- 엔디안: Little Endian
- 문자열: UTF-8, length-prefixed
- 포트: TCP 7777 / UDP 7778
- 좌표계: Unity 표준 (좌수 y-up)

### MVP 패킷 카테고리

**인증 (TCP)**
- `C_Login` (닉네임)
- `S_LoginResult` (success, sessionToken, reason?)

**매칭 (TCP)**
- `S_MatchingStatus` (현재 N/4)
- `S_GameStart` (4명 정보: sessionToken, 닉네임, 팀, 시작 위치)

**인게임 입력/스냅샷 (UDP)**
- `C_Input` (move dir, jump, yaw, pitch, sequenceNumber)
- `S_Snapshot` (4명 위치/회전/HP/사망 플래그/isGrounded)

**인게임 이벤트 (TCP)**
- `C_Fire` (카메라 위치, forward 방향)
- `S_HitResult` (피격자 sessionToken, 데미지, 남은 HP) — 4명 브로드캐스트
- `S_RoundStart` (라운드 번호, 시작 위치 리셋)
- `S_RoundEnd` (승리 팀, 현재 점수)
- `S_MatchEnd` (승리 팀, 최종 점수)
- `C_Quit` (명시적 종료, MVP에서는 사실상 안 씀 — 끊김 감지로 대체)

### 4주차 확장 패킷 (MVP 제외)
- 로비/방: `C_RoomList`, `S_RoomList`, `C_CreateRoom`, `S_RoomCreated`, `C_JoinRoom`, `S_JoinResult`, `C_Ready`, `S_RoomState`, `C_StartGame`
- **탄약/재장전**: `C_Reload`, `S_AmmoUpdate` (또는 Snapshot에 ammo 필드 추가)

## 9. 5주 개발 계획 요약

상세는 별도 WBS (`WBS_HD_Server_v2.md`, `WBS_YG_Client_v2.md`).

### 주차별 목표 요약
| 주차 | 통신 | 틱 모델 | 종료 시점 |
|---|---|---|---|
| 1주차 | TCP | 받자마자 처리 | 4명 자동 매칭으로 자동 인게임 진입 (씬 전환까지) |
| 2주차 | TCP | 받자마자 처리 | 4명이 1인칭으로 움직이고(점프 포함) 사격 + HP 감소 |
| 3주차 | **TCP+UDP** | **30Hz 시뮬 / 20Hz 스냅샷 도입** | 위치/입력 UDP 교체 + 라운드/매치 풀 사이클 ⭐ **(발표 가능 시점)** |
| 4주차 | TCP+UDP | 동일 | 진짜 로비/방 + ESC 모달 + **탄약/재장전** + 봇 풀 기능 |
| 5주차 | TCP+UDP | 동일 | 안정화 + 발표 준비 |

### 위험 시점 점검
- **2주차 말**: TCP 위치 동기화 + 점프 중력 안정화 안 되면 → 3주차 초까지 보강 후 UDP 도입
- **3주차 목요일**: UDP 도입 막히면 → 금요일 폴백 결정 (TCP 유지로 발표 가능 보존)
- **4주차 중**: 진짜 로비 통합 시 깨지면 → 자동 매칭 코드 보존하면서 점진 교체

## 10. 봇 더미 클라이언트 (필수 도구)

**왜 필수인가**: 5주 내내 4인 환경 테스트가 필요한데 매번 사람 4명 모으는 건 불가능. 자동 매칭 모델 덕분에 **봇 4개 띄우면 자동으로 매치 시작** → 테스트 환경 단순화.

### 단계별 기능
| 주차 | 추가 기능 |
|---|---|
| 1 | 서버 접속 + 로그인 (TCP) — 자동 매칭에 자동 편입 |
| 2 | 랜덤 이동 (TCP, 임시) |
| 3 | UDP 이동 + 랜덤 사격으로 교체 |
| 4 | (확장: 로비/방 통합 시 그쪽 흐름 따름, 재장전 입력 추가) |
| 5 | (선택) 다중 봇 동시 실행 → 부하 테스트 |

**부수 효과**: README에 "봇 N개 동시 접속 부하 테스트" 한 줄 → 포트폴리오 가치 +1

## 11. 구현 한계 및 향후 과제 (포트폴리오 핵심)

본 프로젝트는 게임 서버 학습 1단계로, 다음 항목은 **의도적으로 구현하지 않았으며**, README와 면접에서 명시적으로 설명한다.

| 항목 | 미구현 사유 | 향후 학습 방향 |
|---|---|---|
| **Client-Side Prediction** | 5주 범위 초과 | 클라가 입력 즉시 예측 이동 → 서버 응답 시 reconciliation |
| **Lag Compensation** | 학습 1단계 범위 초과 | 서버측 history buffer + 시간 되감기로 hit 판정 |
| **Delta Compression** | 풀 스냅샷으로 충분 | dirty flag 기반 변경분만 송신 |
| **Interest Management** | 4인 규모는 불필요 | 거리 기반 송신 대상 필터링 |
| **매치별 프로세스 분리** | 단일 매치/단일 프로세스로 충분 | Gateway / MatchMaker / GameServer 분리 |
| **다중 매치 동시 진행** | MVP 단일 매치로 단순화 | 매치별 PlayerState 격리, 매치 ID 기반 패킷 라우팅 |
| **MMR/스킬 기반 매칭** | FIFO로 단순화 | 매칭 점수 + 풀에서 비슷한 수준 묶기 |
| **계정/전적 영속화** | DB 미사용 | Redis(세션) + RDB(전적) 도입 |
| **치팅 방지** | 위치/HP는 server authoritative로 자연 차단 | aim assist 등 클라단 치팅은 별도 영역 |
| **헤드샷/바디샷** | hitscan 단순화 (구 1개 근사) | 멀티 히트박스 (머리/몸통/다리) |
| **로비/방 시스템** | MVP 매칭 핵심 보존 | 4주차 확장 또는 방학에서 진짜 로비로 |

이 표 자체가 **"다음 단계가 뭔지 안다"는 증거**로 작동한다.

| 패킷 프로토콜 변경 빈번 | 1주차에 충분히 설계, 변경 시 `Shared` 즉시 동기화 |
| 한쪽 작업 지연 → 통합 막힘 | 매주 금요일 통합 테스트 고정 |
| 4주차 진짜 로비 통합 시 자동 매칭 깨짐 | 1~3주차 자동 매칭 코드 보존하면서 점진 교체 |
| 1~2주차에 정식 틱 루프부터 짜려는 유혹 | **금지**. "받자마자 처리" 단순 모델로 시작, 3주차에 정식 도입 |

## 16. 최종 정리

본 프로젝트는 게임서버프로그래밍 수업 학기 프로젝트이자 **2인 팀의 게임 서버 직무 포트폴리오 산출물의 1단계**다.

- **자체 제작** C# .NET 8 서버 + **Unity 2022 LTS** 클라이언트 + 봇 더미 클라
- **자동 매칭 (FIFO, 단일 매치)**, **TCP + UDP 하이브리드**, **Server Authoritative + State Sync**
- **틱 루프 단계 도입**: 1~2주차 받자마자 처리 → 3주차 30Hz 시뮬 / 20Hz 스냅샷
- **2vs2 / 3D 1인칭 FPS / 1맵 / 1무기 hitscan / 3라운드 2선승**
- **이동 + 점프** (단순 중력 시뮬레이션), **정확한 ray-sphere hitscan**, **무한 탄약 (탄약은 4주차 확장)**
- **Walking Skeleton 전략** (1주차부터 통합, 점진 강화)
- **비주얼은 프리미티브** (캡슐, Line Renderer, 기본 UI) — 시스템 검증 우선
- **학기 = 동작 + 발표, 방학 = 포트폴리오 보강 + C++ 리메이크**
- 코드 + README + 아키텍처 문서 + 패킷 명세 + 시연 영상까지 산출물 포함 (방학 완성)
- **구현하지 않은 항목**(Client Prediction, Lag Compensation, 다중 매치 등)을 명시적으로 정리하여 다음 단계에 대한 이해도까지 함께 어필

화려한 콘텐츠가 아니라 **"게임 서버를 직접 짤 수 있고, 다음 단계가 뭔지 안다"**를 증명하는 것이 최우선 목표다.
