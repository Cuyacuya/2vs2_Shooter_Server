## 변경 이력

- **v1** — 초기 WBS (로비/방 중심)
- **v2** — 기획서 v1.3 반영
  - **MVP를 자동 매칭 기반 풀 사이클로 재정의** — 1~3주차에서 로비/방 제거
  - **3D 3인칭 + 이동 + 점프** — 2주차에 서버측 중력 시뮬 추가
  - **정확한 ray-sphere hitscan** — MVP 필수 (기존 "거리 기반 단순화" 폐기)
  - **30Hz 시뮬 / 20Hz 스냅샷 정식 틱 루프** — 3주차에 도입 (1~2주차는 받자마자 처리)
  - **TCP 끊김 감지 → 사망 처리** — MVP 필수
  - **로비/방 + ESC 모달 + 탄약/재장전** → 4주차 확장으로 통합
  - .NET 8 / Unity 2022 LTS

## 작업 진행 방식 (Workflow)

### 등급별 절차

| 등급 | Claude 역할 | 사용자 절차 |
|---|---|---|
| 🟢 적극 | 완성 코드 제공 | 한 줄씩 읽고 모르는 줄은 질문 → 커밋 (5~10분) |
| 🟡 보조 | 완성 코드 + 라인별 해설 제공 | **읽고 → 가리고 → 재작성** → diff 비교 → 커밋 |
| 🔴 직접 | 개념/다이어그램/함정만 설명 (코드 X) | 30분 타임박스로 직접 작성 → 막히면 단계적 힌트 요청 ("방향 → 의사코드 → 해당 줄") |

### 한 작업씩 끊어가기 원칙
- 한 번에 여러 스크립트를 만들지 않는다. **1 스크립트 작성 → 핵심 개념 학습 → 셀프 퀴즈/실험 → 커밋 → 다음**.
- 학습이 안 된 상태로 다음 단계 진입 금지. "동작은 하는데 왜 그런지 모름" 상태가 누적되면 3주차 틱 루프에서 무너진다.
- 매 커밋 전 본인에게: "이 파일에서 한 줄이라도 설명 못 하는 게 있나?" 체크.

### 셀프 퀴즈 / 실험 패턴 (학습 정착용)
- **셀프 퀴즈**: 작성 직후 Claude가 던지는 질문 4~6개에 답해보기. 막히면 그 부분만 다시 설명 요청.
- **깨뜨려보기 실험**: 일부러 한 줄 바꿔서 어떻게 깨지는지 확인 (예: Serialize 순서 뒤집기). 정상 케이스만 보면 이해 못 함.

### 2주차 ray-sphere 힌트 (학습 포인트)
구체 중심을 C, 반지름 r, ray 시작점 O, 방향 d (단위 벡터)일 때:
- L = C - O
- t_ca = L · d  (음수면 ray 뒤쪽에 구가 있음 → miss)
- d_squared = L·L - t_ca²  (구 중심까지 수선 거리²)
- d_squared > r² → miss
- 그 외 → hit, 거리 = t_ca - sqrt(r² - d_squared)

이 식은 직접 유도/구현해볼 것. AI에 통째로 받지 말 것 (🔴).

### 🎯 3주차 끝 = 발표 가능 시점 (무조건 사수)

### 3주차 핵심 학습 (가장 학습 가치 큼)
- **UDP의 비연결 특성** — sessionToken으로 발신자 매칭하는 이유 직접 체감
- **TCP vs UDP 직접 비교** — 같은 위치 동기화를 두 번 짜며 차이 체감
- **정식 틱 루프** — Stopwatch 기반 일정 주기 처리, 입력 큐잉, 스냅샷 다운샘플
- **시뮬 빈도 vs 송신 빈도 분리** — 정확도 vs 대역폭 tradeoff
- **메시지 분류 원칙** — 신뢰성(TCP) vs 자주 보냄(UDP)

### 3주차 위험 시점 ⚠️ (목요일 점검)
**UDP 도입 또는 정식 틱 루프가 막혀서 목요일 끝나도 안 됨:**
- 금요일 결정: TCP+받자마자 처리 유지로 폴백 vs 토요일 + 다음 주 추가 작업
- 폴백 시: 2주차 코드 그대로 사용, UDP/틱 루프는 5주차 보강 또는 방학으로
- 발표는 가능 (TCP+받자마자 처리로도 동작은 함)

**둘 중 더 위험한 건 정식 틱 루프**. UDP는 "패킷 받기"만 하면 일단 동작하지만, 틱 루프는 입력 큐잉/스냅샷 다운샘플/시간 보정까지 묶여 있어 한 군데 틀리면 전체가 망가짐.

### 3주차 종료 기준 ✅
- [ ] 위치 동기화가 UDP로 동작 (TCP보다 부드럽다는 체감)

- [ ] 라운드/매치 종료 = TCP로 안정적
- [ ] 한 팀 2승 → 매치 종료 → 클라 메인 복귀

⚠️ **이 주차는 4주차 확장 기능이 한꺼번에 들어가서 빡빡하다.** 시간 부족하면 우선순위는: 로비/방 > 탄약/재장전 > 봇 풀 기능. ESC 모달은 클라 단독 작업이라 서버 영향 적음.

### 4주차 핵심 학습
- **상태 머신 확장** — Idle 1개 → Lobby + RoomWaiting + InGame 다중 상태로 확장
- **점진 교체 패턴** — 1~3주차 임시 로직 보존하며 정식 로직으로 교체 (Walking Skeleton의 정수)
- **타이머 기반 처리** — 재장전 endTime 비교 패턴 (틱 루프와 결합)

### 4주차 위험 시점
- 수요일까지 진짜 로비 통합 안 되면 → 목요일 탄약/재장전 빼고 로비/방 마무리 우선
- 자동 매칭 코드를 너무 일찍 지우면 안 됨 → 양쪽 다 동작하는 상태에서 점진 교체

## 서버 담당이 자주 빠지는 함정

| 함정 | 결과 | 대응 |
|---|---|---|
| TCP 프레이밍 안일하게 처리 | 패킷 합쳐짐/쪼개짐에 깨짐 | 1주차에 length-prefixed 정확히, 비정상 케이스 단위 테스트 |
| 동시성 lock 누락 | 4명 접속 시 random crash | PlayerState 접근하는 모든 메서드에 lock 적용 (또는 ConcurrentDictionary) |
| 1~2주차에 정식 틱 루프부터 짜려는 유혹 | 1~2주차가 다 날아감 | **금지**. 받자마자 처리로 시작, 3주차에 정식 도입 |
| 점프 isGrounded 깜빡 | 무한 점프 가능 | 지면 접촉 플래그 검사 후 점프 입력 수락 |
| ray-sphere 음수 t 처리 누락 | 등 뒤의 적도 맞음 | t_ca < 0이면 miss 처리 |
| TCP 끊김 감지 누락 | 매칭 큐에 좀비 세션 누적 | ReadAsync 예외/0 byte 받으면 정리 |
| Shared 동기화 깜빡 | 클라 빌드 깨짐 | 본인이 패킷 추가하면 즉시 클라쪽도 복사 |
| 4주차에 자동 매칭 코드부터 지움 | 통합 테스트 깨짐 | 양쪽 다 동작하는 상태에서 점진 교체 |

## 절대 잊으면 안 되는 것

1. **3주차 끝 = 발표 가능 시점.** 무조건 사수.
2. **1~2주차는 받자마자 처리로 단순하게.** 3주차에 정식 틱 루프로 진화.
3. **Shared 동기화 매번 확인.** 본인 패킷 추가 시 클라쪽도 같이.
4. **봇 클라 1주차부터 작업.** 4인 환경 테스트의 유일한 수단.
5. **점프 중력 시뮬은 콘솔 단위 테스트로 분리 검증.** 게임에 통합한 후엔 디버깅 헬.
6. **ray-sphere hitscan은 직접 유도해서 짤 것.** AI에 통째로 받지 말 것 (학습 핵심).
7. **클라 담당과 매주 금요일 통합 테스트 고정.**
8. **자동 매칭 코드를 4주차에 함부로 지우지 말 것.** 양쪽 다 동작 상태에서 점진 교체.
